# Recurrence Principle

This document records the standing AoA principle of recurrence for long-horizon agentic work.

It does not create a new layer. It does not replace the rootline, the method spine, or the federation rules. It names the recovery law that governs how AoA recovers continuity when a model, route, or runtime seam loses the axis of the work.

## Core law

The AoA recurrence principle is:

1. **when a route loses its axis, it returns before it continues**
2. **return goes to the last valid anchor, not to ambient continuity**
3. **re-entry must be explicit, reviewable, and bounded**
4. **return must preserve source ownership rather than blur it**

## Why the center names this law

AoA is built for long-horizon work across models, layers, and handoffs.

In that terrain, continuity fails in practice:

- small models lose the thread sooner
- larger models may hide loss of thread behind fluent continuation
- longer context alone does not guarantee a trustworthy route
- retries without anchor discipline often multiply drift instead of repairing it

AoA therefore treats recurrence as a federation law rather than as a prompt trick, a vendor feature, or an informal operator habit.

## Recurrence vocabulary

Use this compact vocabulary when naming recurrence posture across the federation:

- `axis` = the current combination of goal, phase, owning source, expected artifact, and proof boundary that makes the next step legitimate
- `drift` = loss, blurring, or substitution of that axis
- `anchor` = the last reviewable object that still preserves enough of the axis to justify re-entry
- `return` = the act of stepping back to that anchor when the current route stops being trustworthy
- `re-entry` = the explicit next step taken from the anchor
- `safe stop` = the honest halt used when no bounded re-entry is yet justified

Recurrence names the standing law. Return names the concrete recovery move inside that law.

## Valid anchors

A valid return anchor should be one or more of these:

- the current route decision or owning navigation object
- the current role contract or handoff boundary
- the current artifact contract or expected output class
- the last validated checkpoint or memory surface
- the source-owned document, object, or authority surface being acted on
- the active proof boundary, approval gate, or escalation boundary

The point of return is not to go backward in the abstract. The point is to recover the smallest reviewable object that can support a legitimate next move.

## What recurrence is not

Recurrence is not:

- a blind retry
- replaying the entire transcript as if more raw context were the same as recovery
- silent context inflation
- generic repetition or cycle doctrine
- scenario-level method by default
- a routing layer pretending to own source meaning
- an infrastructure layer quietly authoring policy
- a derived or retrieved surface replacing authored or core truth

If the system cannot name the anchor, the ownership boundary, and the re-entry step, it is not operating under recurrence. It is improvising continuity.

## Ownership chain

Keep the owning roles explicit:

- `Agents-of-Abyss` owns the federation-level law, vocabulary, and anti-collapse boundary
- `aoa-techniques` may hold reusable recovery practices when they become stable practice objects
- `aoa-skills` may hold bounded execution-time return procedures inside skill contracts
- `aoa-routing` may point to re-entry surfaces and route back toward the smallest valid next object, but it does not own recurrence meaning
- `aoa-memo` owns checkpoint, recall, and provenance surfaces used for return
- `aoa-agents` owns role-contract and handoff posture for return between actors, tiers, or operator states
- `aoa-playbooks` owns scenario-level return routes, fallback posture, reroute posture, and escalation posture once return becomes recurring method
- `aoa-evals` owns bounded proof surfaces for drift detection, recovery quality, and failure honesty
- `abyss-stack` owns runtime wrapper and context-rebuild implementation beneath the doctrine
- `Tree-of-Sophia` remains the source authority whenever recurrence crosses into ToS-authored knowledge surfaces

## Relation to existing center doctrine

This principle is consistent with the current AoA center law:

- the rootline already favors self-agent checkpoint posture before autonomy rhetoric
- the method spine already says recurring multi-layer method belongs in playbooks
- the federation rules already require explicit ownership, explicit bridges, and anti-collapse discipline

The recurrence principle does not replace those doctrines. It gives them a recovery law.

## Practical review rule

A recurrence-ready surface should answer these questions clearly:

- what axis was lost?
- what anchor is being trusted?
- who owns that anchor?
- what minimal context, artifact, or memory object survives the return?
- what proof or review gate checks the re-entry?
- when does return become reroute, escalation, or safe stop?

Return is the bounded step back to a valid anchor. Reroute chooses a different next route. Escalation hands responsibility to a stronger actor or review surface. Safe stop halts honestly when no bounded re-entry is yet justified.

If those answers are not explicit, the route is still depending on informal continuity rather than a public recurrence posture.

## Anti-collapse rule

The recurrence principle is valid only when it preserves the federation's existing boundaries:

- source repositories still own meaning
- routing stays navigational
- memory stays explicit and reviewable
- scenario-level method stays in playbooks when it becomes recurring
- runtime implementation stays downstream of doctrine
- recurrence remains reversible and bounded rather than turning the whole system into one ambient conversation

## Compact law

When AoA loses the axis of a route, it should not continue by momentum.

It should return to the last valid anchor, preserve ownership, and re-enter through an explicit bounded step.
