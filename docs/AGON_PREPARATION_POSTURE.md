# Agon Preparation Posture

This note captures the pre-Agon landing boundary for the AoA center.
It prepares the ground for Agon without starting Agon, creating a new sibling
repository, or claiming that a runtime protocol already exists.

## Current status

Agon is not a live implementation layer yet.
Agon is not a new sibling repository.

The current honest posture is preparation:

- keep the future Agon center-owned inside `Agents-of-Abyss`
- keep it central but not total
- keep `Tree-of-Sophia` as slow source-first canon, not an arena
- keep `aoa-sdk` and `abyss-stack` as control-plane and runtime substrate, not
  sources of Agon law
- keep neighboring AoA repositories as owners of their specific meaning
- avoid minting a new `Agon-of-Abyss` sibling repository before the center has a
  bounded protocol kernel

## Source-of-truth split

The future Agon contour should use this ownership map:

| concern | owning home | boundary |
|---|---|---|
| Agon law, session lifecycle, lawful moves, contradiction ledger, terminal forms, and promotion discipline | `Agents-of-Abyss` | center-owned constitutional protocol, not runtime service truth |
| agonic and assistant actor form, role contracts, office/kind split, rights, refusal posture, and role-facing eligibility | `aoa-agents` | role-owned shape only; no scar ledger, verdict, runtime packet canon, or ToS canonization |
| reusable thinking and pressure-handling practice | `aoa-techniques` | technique canon only |
| lawful bounded moves that agents may execute | `aoa-skills` | execution workflow canon only |
| arena entry hints and `agon_needed`-style dispatch | `aoa-routing` | thin routing gate only, not arena sovereignty |
| trial rituals, repeated scenarios, and early mechanical arena campaigns | `aoa-playbooks` | recurring scenario method only |
| lawful session verdicts, breach checks, and promotion-worthiness proof | `aoa-evals` | proof and judge layer only |
| scars, delta history, retention checks, and recall of arena outcomes | `aoa-memo` | memory truth only |
| derived movement summaries such as delta depth, contradiction integrity, summon economy, closure discipline, prediction accuracy, and retention strength | `aoa-stats` | derived observability only |
| derived survivor structures and retrieval-ready lifts | `aoa-kag` | derived substrate only |
| typed exchange helpers, discovery, and bounded orchestration support once a real protocol exists | `aoa-sdk` | control-plane support only |
| durable runtime body once Agon becomes a service contour | `abyss-stack` | runtime implementation only |
| slow source-linked canonization after reviewable survival | `Tree-of-Sophia` | no direct arena write path |

## Pre-Agon wave order

The preparation order is:

1. Form agent kinds in `aoa-agents`: agonic actors and assistant actors must
   be distinguishable before the arena opens.
2. Give those actors reusable practice and lawful moves through
   `aoa-techniques` and `aoa-skills`.
3. Teach `aoa-routing` to recognize when service handling is insufficient and
   when an arena gate may be needed.
4. Land a narrow center-owned Agon kernel in `Agents-of-Abyss`: duel first,
   mechanical profile first, typed charter first.
5. Add witnesses, verdicts, scars, and derived summaries through `aoa-evals`,
   `aoa-memo`, and `aoa-stats`.
6. Package repeated early trials in `aoa-playbooks`.
7. Only after those source-owned contracts are stable should `aoa-sdk` and
   `abyss-stack` grow typed control-plane and runtime support.
8. Only after retention, proof, and source-first review should any survivor
   approach `Tree-of-Sophia` as a candidate.

## Stop-lines

Do not:

- treat Agon as a chat style, prompt pack, or RPG decoration
- treat Agon as a new sibling repository
- create a new sibling repository before the center-owned protocol has a
  bounded first kernel
- put arena law in `aoa-routing`, `aoa-sdk`, or `abyss-stack`
- let `Tree-of-Sophia` become a runtime conflict arena
- let assistants become hidden agonic contestants by drift
- write scars, deltas, or retention history into role profiles
- treat an arena win as ToS canonization
- claim a live Agon runtime before protocol, proof, memory, routing, and
  runtime contracts actually exist

## Readiness test

The next Agon move is ready to start only when the first patch can honestly say:

- it is center-owned and bounded
- it does not create a new repo
- it starts with a duel kernel or equally narrow protocol surface
- it names the neighboring owner handoffs rather than absorbing them
- it has validation or explicit review checks
- it preserves the path `arena -> memo -> eval -> kag -> ToS candidate`
  without skipping directly into canon

Until then, the correct work is preparation, not activation.
