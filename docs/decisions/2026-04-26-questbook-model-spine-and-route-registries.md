# Decision Note: Questbook Keeps A Model Spine And Registry-Backed Route Tables

Status: accepted
Date: 2026-04-26

## Context

Questbook grew from a public obligation index into lane-first source objects,
generated read models, relation metadata, and ready owner-route maps.

The original model document was becoming a mixed surface: lifecycle law,
execution passport, harvest thresholds, and owner-route table behavior all
competed for attention in one place.

The Experience ready owner-route table also had a second risk: hand-edited
Markdown rows could drift from `quests/experience/ready/`, the owner-request
queue, or `mechanics/experience/OWNER_REQUESTS.md`.

## Options considered

1. Keep all detail in the Questbook model document.
2. Split model detail into narrow Questbook source documents, but keep route
   tables hand-edited.
3. Keep the model as a spine, split detail into narrow part packages, and
   generate ready owner-route tables from registries.

## Decision

AoA keeps `mechanics/questbook/parts/model-spine/README.md` as the Questbook
model spine.

Detailed law now lives in narrow part packages:

- `mechanics/questbook/parts/lifecycle-law/`
- `mechanics/questbook/parts/execution-passport/`
- `mechanics/questbook/parts/harvest-route/`
- `mechanics/questbook/parts/lane-owner-routes/`
- `mechanics/questbook/parts/relation-shape/`

Ready owner-route tables are registry-backed. The first registry is:

- `mechanics/questbook/parts/lane-owner-routes/experience-ready-owner-routes.json`

Its Markdown projection is generated by:

- `mechanics/questbook/scripts/build_ready_owner_routes.py`
- `mechanics/questbook/scripts/validate_ready_owner_routes.py`

## Rationale

- Entry surfaces stay readable for small-context agents.
- Source law remains package-local and reviewable.
- Route rows can be checked against ready quest files and owner-request IDs.
- Generated route tables stay useful for humans without becoming authoring
  surfaces.

## Consequences

- new Questbook law should usually land in the narrow source doc that owns the
  question, not by expanding the spine
- ready-route Markdown projections must be rebuilt from registries
- route registries do not update owner-request status, prove owner acceptance,
  or close quests
- a second lane registry should be added only when that lane has real ready
  owner-route pressure

## Source surfaces

- `mechanics/questbook/parts/model-spine/README.md`
- `mechanics/questbook/parts/lifecycle-law/`
- `mechanics/questbook/parts/execution-passport/`
- `mechanics/questbook/parts/harvest-route/`
- `mechanics/questbook/parts/lane-owner-routes/`
- `mechanics/questbook/parts/relation-shape/`
- `mechanics/questbook/parts/lane-owner-routes/experience-ready-owner-routes.json`

## Follow-up route

Route future lane-route growth through `mechanics/questbook/parts/lane-owner-routes/`
and its builders before adding hand-authored route tables.
