# VIA_NEGATIVA

Via negativa is the subtraction posture for AoA.

The ecosystem already knows how to add.
Waves 1 through 4 added doctrine, receipts, proof, routing hints, memory
objects, handoff envelopes, playbook lanes, projection health signals, and
derived summaries.
That growth was useful.
It also created the exact moment when a pruning discipline becomes necessary.

This document defines the subtraction posture for the federation.

## Core claim

When the system grows, fragility often arrives disguised as convenience.

It appears as:

- duplicated doctrine
- helper surfaces that quietly seize policy
- active runbooks polluted with archived matter
- derived layers that sound canonical
- memory that acts like proof
- projections that start outranking sources
- more profiles, bundles, and lanes than the distinctions can justify

Via negativa says:
remove those exposures before inventing new protective structure.

## Pruning order

When a surface looks suspicious, use the lightest action that restores a clear
boundary:

1. keep
2. merge
3. move
4. suppress
5. quarantine
6. deprecate
7. delete

Deletion is real, but it is last.
Most fragility in documentation and workflow systems is cured earlier by
putting the right thing back in the right home.

## What subtraction protects

Via negativa is not anti-structure.
It protects a specific set of structures that should survive stress:

- source-owned meaning
- provenance anchors
- explicit receipts
- bounded proof surfaces
- derived summaries with clear suppression rules
- thin routing
- reviewable memory
- explicit handoff and re-entry conditions
- explicit mutation boundaries
- clear public source-of-truth documents

## Negative rules

The federation should increasingly express itself through rules about what
layers must not do:

- the center must not replace owner repos
- proof must not replace operations
- stats must not replace truth
- routing must not replace policy
- memo must not replace proof
- KAG must not replace sources
- SDK must not replace ownership
- playbooks must not replace agent contracts
- product edges must not hide mutation

These prohibitions are not dead weight.
They are the rails that keep future growth from becoming sludge.

## One-in-one-out posture

Whenever a new tracked surface is proposed, ask first:

- which existing surface becomes redundant if this lands?
- can the new content merge into a canonical file instead?
- is the distinction material enough to survive review six months later?
- if this is a convenience surface, what authority does it risk stealing?
- if it is a temporary bridge, where is its expiry?

If the answer is "nothing becomes redundant", that is a warning siren, not a
triumph.

## Machine-readable candidate posture

`DELETION_CANDIDATES.json` is an inspect-first prompt surface, not a deletion
verdict.
It exists to make pruning reviewable and owner-aware.
Any concrete merge, move, suppress, quarantine, deprecate, or delete action
still belongs to the owning repository and must preserve provenance, receipts,
and bounded proof.

## Non-goals

Via negativa does not mean:

- erase useful history
- remove validated evidence because it feels messy
- centralize cleanup power in one layer
- automate deletion before the ecosystem has stable pruning instincts
- turn subtraction into aesthetic minimalism for its own sake

The target is not emptiness.
The target is a system that remains legible, bounded, and harder to misread
after every growth cycle.
