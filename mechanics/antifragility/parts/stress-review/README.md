# Stress Review

Stress review turns failure, degraded mode, cleanup pressure, or operator pain
into a bounded improvement route.

## Use When

- A failure or stress event exposed a fragile seam.
- A degraded mode continued and needs review.
- A cleanup proposal claims the system became stronger.
- A future similar event should be handled better.

## Do Not Use When

- The owner repository has not named the event or receipt.
- The work is only aesthetic cleanup.
- The next step is already a proof verdict, memory object, or runtime fix.

## Route Check

Name the stress, where it was bounded, what evidence exists, what degraded mode
or stop condition applied, and which owner must carry the repair.

## Active Outputs

- Stress review note.
- Repair request.
- Evidence request.
- Degraded-mode stop-line.
- Adaptation route.

## Next Route

Route repair to the owner repository, proof to `aoa-evals`, retained lessons to
`aoa-memo`, and recurring response choreography to `aoa-playbooks`.
