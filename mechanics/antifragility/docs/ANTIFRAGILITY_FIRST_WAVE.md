# ANTIFRAGILITY FIRST WAVE

## Aim

Land the smallest cross-repo wave that makes antifragility concrete instead of merely admirable.

## Golden family

Wave 1 anchors on one concrete stressor family:

**hybrid-query degradation in `ATM10-Agent` where retrieval succeeds but the KAG stage fails or returns no useful expansion.**

This family is strong enough to exercise:

- bounded degraded continuation
- source-owned receipts
- later adaptation deltas
- split-axis evals
- derived vector summaries

## Repositories in scope

### `Agents-of-Abyss`
Own the doctrine and first-wave boundaries.

### `ATM10-Agent`
Own the first concrete source receipts and adaptation deltas.

### `aoa-techniques`
Own the portable practice patterns.

### `aoa-evals`
Own the bounded proof bundle for antifragility posture.

### `aoa-stats`
Own the derived vector view.

### `aoa-skills`
Own a lightweight authoring addendum so fallback and receipt expectations can spread cleanly into future skills.

## Deliverables

### P0
- doctrine doc
- first-wave scope doc
- source-owned stressor receipt schema
- source-owned adaptation delta schema
- examples for the golden family
- two portable techniques
- one bounded eval bundle
- one eval report schema
- one derived antifragility vector schema
- one stats doctrine doc

### P1
- repo-native builders, validators, and tests
- broader adoption in routing, memo, sdk, and runtime surfaces
- promotion of shared source-owned contracts if multiple repos converge

## Completion criteria

Wave 1 is meaningfully landed when all of the following are true:

1. the center doctrine is stable enough to quote from downstream repos
2. `ATM10-Agent` can emit or at minimum clearly specify owner-local stressor receipts for the golden family
3. the eval layer can assess split axes without stealing workflow ownership
4. the stats layer can publish a derived vector without becoming a sovereign score layer
5. future repos have a portable pattern to follow

## Non-goals

Wave 1 should not:

- invent a universal hidden runtime
- require a new endpoint for every receipt or recovery surface
- turn all failures into one metric
- rewrite unrelated repos before the first family proves useful
