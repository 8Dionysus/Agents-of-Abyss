# Fragility Blacklist

This blacklist names patterns that should be removed, merged, narrowed, suppressed, quarantined, or refused before the ecosystem grows new surface area.

It is a compact root index. Canonical subtraction doctrine lives in [VIA_NEGATIVA](docs/VIA_NEGATIVA.md). The old deletion-candidate packet is preserved through [audit provenance](../audit/PROVENANCE.md).

| Pattern | Fragility | Default response |
|---|---|---|
| Authority drift | a non-owner layer starts sounding canonical | shrink it back to a hint, view, recall surface, or link |
| Duplicate meaning | two or more tracked surfaces say the same thing with slightly different wording | keep one owner surface and merge or delete the rest |
| Hidden mutation | a helper, route, playbook, or runtime surface performs state change without explicit reviewable posture | stop, expose the mutation boundary, or delete the convenience wrapper |
| Archived material in active surfaces | historical, rollback, blocked, or restore-only guidance appears inside active runbooks or default product docs | move it to archive or delete stale remnants |
| Score sovereignty | a derived or evaluative layer compresses broad signals into one prestige number | split back into vectors, bounded verdicts, or suppressed summaries |
| Unsourced routing | a route hint or dispatch rule depends on anecdote, memory-only recall, or vibes | require owner receipts or bounded proof, or suppress the hint |
| Memory as proof | memo objects speak as if they are live truth rather than reviewed context | downgrade to recall context, add review posture, or remove the object |
| Derived substrate as source replacement | KAG or other generated projections duplicate source text and then begin to outrank it | quarantine stale packs and delete low-value copies |
| Bundle proliferation | evals, techniques, skills, playbook lanes, or agent profiles multiply faster than their distinctions | merge near-duplicates and keep only materially different variants |
| Surface inflation | a repo gains new tracked docs because a thought felt important, not because ownership demanded a new canonical role | merge into an existing canonical surface or keep it local-only |
| Wrapper policy capture | SDK or runtime helpers quietly embed workflow policy that belongs in source repos, routing registries, or explicit owner docs | narrow the helper or remove the wrapper |
| Low-evidence permanence | a weak signal, temporary finding, or thin experimental output becomes a stable tracked surface | suppress, quarantine, or mark expiring |
| Unbounded retry and auto-repair | runtime or orchestration logic keeps intervening without clear stop conditions or operator-visible receipts | add explicit stop thresholds or remove the self-healing shortcut |
| Persona inflation | new agent or role surfaces are created for style, flavor, or slightly different wording rather than real contract change | merge profiles and keep one explicit role contract |
| Exception without sunset | a one-off override becomes permanent because nobody remembers why it exists | add expiry and review, or delete it |

## Use with root cleanup

If the fragile object is in the repository root, apply [docs/ROOT_SURFACE_LAW](../../docs/ROOT_SURFACE_LAW.md) first. The first question is not "is this interesting?" It is "does this object still deserve civic root visibility?"
