# Antifragility Owner Map

Antifragility routes stress and subtraction. It does not execute owner-local
cleanup.

| Concern | Center role | Stronger owner | Stop-line |
|---|---|---|---|
| Stress doctrine | Names the review questions and stop-lines | Owner repository | A stress note is not a repair receipt. |
| Degraded continuation | Names bounded posture requirements | Runtime or product owner | Degraded mode must not be secretly broader than normal mode. |
| Subtraction | Names pruning order and provenance duties | Owner repository | The center must not delete owner truth. |
| Authority drift | Names anti-authority rules | Owning layer | Convenience must not become policy. |
| Fragility patterns | Maintains compact pattern vocabulary | Owner repository plus proof/memory owners | Pattern presence is not a health score. |
| Repair proof | Routes evidence requirements | `aoa-evals` | A cleanup story is not proof. |
| Retained lessons | Routes lessons after review | `aoa-memo` | Memory is not proof or live truth. |
| Derived visibility | Names vector posture | `aoa-stats` | Derived summaries must not reign. |
| Recurring cleanup | Names handoff expectations | `aoa-playbooks` | The center must not run cleanup choreography. |
| Reusable practice | Names candidate practice shape | `aoa-techniques` | Doctrine is not reusable technique canon. |
| Executable workflow | Names execution boundary | `aoa-skills` | A route is not an executable skill. |
| Runtime recovery | Names stop-lines only | `abyss-stack` | Runtime self-healing requires runtime-owner gates. |

## Owner handoff rule

If an antifragility finding needs action outside AoA, create or update an owner
request. Do not mark the request accepted until the owner repository produces a
receipt.
