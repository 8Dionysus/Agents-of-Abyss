# Component Refresh

Component Refresh routes drifting internal technical surfaces through owner
refresh law, reviewed decisions, receipts, and derived summaries.

## Use When

- A named component has stale generated, projected, or contract surfaces.
- Repeated manual patching shows one owner-owned refresh route is needed.
- A reviewed refresh decision must precede derived summary closure.

## Do Not Use When

- The center wants to regenerate owner surfaces itself.
- The signal is only a fixture and no real owner drift exists.
- The route is actually shared-root rollout, generic repair, or release support.

## Route Check

Ask whether there is one named component, one owner repo, one refresh law, one
rollback anchor, and one proof route. If not, keep the signal as a request, not
a refresh claim.

## Active Outputs

- component refresh request
- owner refresh route
- drift hint boundary
- reviewed decision route
- receipt and summary route
