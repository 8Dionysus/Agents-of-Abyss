# Checkpoint Intake

Checkpoint Intake reads provisional carry and decides whether the cycle has
enough reviewed evidence to move.

## Use When

- Work resumes from a checkpoint, pause, hook note, closeout residue, or return
  anchor.
- The evidence is useful but still lower-authority than owner-local truth.
- The next stage needs a named owner before mutation continues.

## Do Not Use When

- The evidence is already reviewed closeout.
- The owner repository already accepted the object.
- The route needs memory canon, runtime activation, or proof verdict.

## Route Check

Ask what the checkpoint can honestly support. If it only preserves context,
keep it provisional. If it names a reviewed next step, route to closeout,
harvest, diagnosis, repair, quest, or owner followthrough.

## Active Outputs

- checkpoint route
- provisional evidence refs
- next-stage hint
- owner hint
- review-needed note
