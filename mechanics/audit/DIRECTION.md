# Audit Direction

Audit now serves as the center route for reviewable seeing. It gives agents a lightweight way to name what is visible, what is evidenced, what is risky, what finding can be carried, and which owner should receive the next move.

## Current Contour

- Keep active audit docs short, route-shaped, and owner-aware.
- Preserve old audit material in `legacy/raw/` with a controlled bridge through `PROVENANCE.md`.
- Use part contracts for the working grammar instead of importing old protocols into everyday reading paths.
- Keep proof verdicts, remediation, release support, runtime authority, memory truth, generated truth, and source-authored meaning with stronger owners.

## Working Route

1. Start with the source surface and owner boundary.
2. Build the evidence ledger before naming a finding.
3. Separate risk signal from proof verdict.
4. Route each finding to the owner that can actually act.
5. Use the validation gate to report what was checked, skipped, and still risky.
6. Promote repeated audit shapes through owner requests, playbooks, skills, techniques, evals, memo, or stats when the pattern stabilizes.

## Stop-lines

- Audit may reveal and route; it does not silently fix the owner surface.
- Audit may name evidence quality; it does not certify proof strength.
- Audit may preserve legacy receipts; it does not make archived text active law.
- Audit may expose generated or derived signals; it does not let them author meaning.

## When Time Has Come

- Add an artifact map only when audit schemas, examples, scripts, or tests grow beyond the current package-level validator.
- Add deeper campaign parts only when repeated audit campaigns appear in multiple owner repos.
- Add owner-local audit templates only after sibling repos land their own audit posture.
