# Boundary Bridge Direction

Boundary bridge exists because the ecosystem needs contact between owners
without collapsing ownership.

The mechanic is not a generic connector. It is a restraint surface for moments
where a route, proof, projection, analogy, witness, public claim, or handoff
must pass near another owner-owned truth.

This file owns the current operating direction only. It does not replace the
part map, landing ledger, roadmap, owner-request packet, owner map, or
provenance bridge.

## Source-of-truth split

- `README.md`: package entry card and shortest route.
- `DIRECTION.md`: current operating direction.
- `PARTS.md`: active part map.
- `parts/`: concise bridge contracts.
- `OWNER_MAP.md`: bridge owners and modes.
- `OWNER_REQUESTS.md`: center-side owner request packet.
- `LANDING_LOG.md`: canonical landing ledger.
- `ROADMAP.md`: future contour, not a historical ledger.
- `PROVENANCE.md`: the only active bridge back to source accounting.
- `docs/`: active source doctrine for ToS-support boundaries.
- `legacy/`: preserved raw receipts when future bridge packets need them.

## Current direction

Make crossings explicit:

- name the source owner
- name the receiving or derived owner
- name the bridge mode
- name what must not transfer
- name the proof or receipt needed before the bridge may be claimed as landed

## Current center posture

The first mature bridge family is ToS support. It teaches the hardest rule:
AoA may support, witness, derive, route, or prepare around ToS, but AoA does
not author ToS meaning.

## Growth rule

A new bridge part belongs here only when it prevents repeated boundary drift.
If a one-off handoff can stay in `OWNER_REQUESTS.md`, do not create a new part.

## Stop-line

If the bridge makes ownership less clear, it is not a bridge. It is boundary
pollution and should be returned to the owner map before more text is added.
