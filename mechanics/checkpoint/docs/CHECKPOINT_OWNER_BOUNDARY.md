# Checkpoint Owner Boundary

This note keeps checkpoint from collapsing owner layers.

## Center boundary

`Agents-of-Abyss` may define checkpoint law, vocabulary, owner map, stop-lines,
and route grammar.

It may not implement checkpoint controls, store runtime truth, promote memory
objects, issue proof verdicts, run scenarios, route live traffic, or publish
stats authority.

## Owner-local boundary

Every owner repository may express checkpoint according to its own lane:

- SDK: controls and helper surfaces.
- Skills: protocol and explicit bridge execution.
- Agents: actor-side checkpoint posture.
- Memo: relaunch and writeback artifacts.
- Playbooks: scenario choreography.
- Evals: proof and regression reading.
- Routing: re-entry hints.
- Stats: derived visibility.
- Dionysus: reviewed snapshots and seed lineage.
- Runtime: export and receipt plumbing.

## Boundary failures

Treat these as failures:

- center docs claiming a sibling implementation is operational
- SDK helpers becoming doctrine
- raw checkpoint notes becoming durable memory
- routing hints interpreting checkpoint meaning
- stats summaries becoming proof
- runtime exports bypassing memo or eval boundaries
- owner requests being worded as owner acceptance

## Repair route

When boundary failure appears:

1. Return to [OWNER_MAP](../OWNER_MAP.md).
2. Name the stronger owner.
3. Convert the center claim into an owner request or stop-line.
4. Leave implementation truth in the owner repository.
