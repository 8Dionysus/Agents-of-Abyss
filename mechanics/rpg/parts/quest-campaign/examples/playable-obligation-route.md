# RPG Playable Obligation Worked Route

Worked route, not a template. Do not copy this into every quest.

This example shows how an ordinary repository task can become readable as a
quest-shaped obligation and then receive an RPG reading without moving source
truth away from Questbook, the owner repository, or the proof owner.

## Ordinary Task

Prepare the first body-facing RPG runtime/projection slice so RPG contour data
can become readable to services and frontend layers without mutating upstream
meaning.

Plain reading:

- the work belongs to `Agents-of-Abyss` until an owner-local projection request
  is accepted
- the source object is already a Questbook item, not an RPG-owned quest
- runtime behavior still belongs to `abyss-stack`
- proof must show that source keys, owner boundaries, and generated routes stay
  intact

## Quest Route

Source quest:

- `quest_ref`: `quests/center/triaged/AOA-Q-0008.yaml`
- `lane`: `center`
- `state`: `triaged`
- `owner_surface`: use the value in the source quest; do not copy historical
  route labels into active RPG docs
- `anchor_ref`: `mechanics/rpg/parts/runtime-projection/README.md`

Questbook reading:

- keep lifecycle state, lane placement, relation shape, and source contract in
  Questbook
- do not promote, close, or reanchor the quest from RPG language
- use the quest source to name the durable obligation before attaching any
  campaign, party, stake, unlock, or consequence reading

## RPG Reading

Derived playable reading:

- `quest_ref`: `quests/center/triaged/AOA-Q-0008.yaml`
- `owner_route`: `mechanics/rpg/parts/runtime-projection/README.md` first,
  then `abyss-stack` only after an accepted owner-local projection request
- `campaign_ref`: first RPG runtime body contour
- `party`: RPG mechanic steward plus runtime owner when the handoff opens
- `stake`: RPG grammar can become service-readable without becoming source
  doctrine
- `proof_route`: RPG and Questbook validators named in the relevant `AGENTS.md`
  files, followed by the repository release check when generated or route
  surfaces change
- `unlock_question`: whether a runtime projection adapter can be requested,
  not whether RPG can declare runtime behavior
- `consequence_route`: update RPG and Questbook landing logs only after checked
  landing; route memory-shaped consequence to `aoa-memo`

The RPG reading makes the obligation more playable because it names party,
stake, proof, unlock pressure, and consequence. It still does not own the quest
object, runtime state, proof verdict, or owner acceptance.

## Proof Route

Before claiming movement, the agent must prove the plain route first:

- the quest source still names the durable obligation and its lane
- RPG usage still passes the "adds routing, judgment, memory, proof, or
  consequence" test
- Questbook still owns lifecycle state and generated read models
- runtime projection remains an owner handoff, not center activation
- release-facing checks pass if generated, registry, or route surfaces changed

If any proof step fails, remove the RPG reading and fix the plain Questbook or
owner route first.

## Owner Handoff

Handoff shape:

- RPG can say why this is a playable obligation and what the stake is
- Questbook can say where the obligation lives and whether its lifecycle moved
- `abyss-stack` decides whether a runtime projection contract can land
- `aoa-evals` or owner-local proof decides whether the movement is accepted
- `aoa-memo` may preserve consequence only after reviewed evidence exists

Closeout should say:

- changed source object, if any
- changed RPG reading, if any
- proof route used
- owner request opened or left untouched
- what must not be claimed yet

## Stop-lines

- Do not copy this structure into every quest.
- Do not add an RPG reading when plain Questbook fields already answer the next
  move.
- Do not claim quest closure, proof completion, runtime activation, owner
  acceptance, reward authority, or memory ownership from this example.
